Quando comecei a estudar programação, imaginava que desenvolver software era simplesmente escrever código.

Com o tempo descobri que código é apenas uma pequena parte da construção de um sistema.

A verdadeira complexidade está em entender problemas, levantar requisitos, definir arquitetura, integrar tecnologias e criar soluções que realmente gerem valor para as pessoas.

Foi exatamente isso que aprendi durante o desenvolvimento do **HashDocs**.

👉 Projeto: https://www.hashdocs.com.br/

Mais do que um projeto de blockchain, o HashDocs se tornou uma escola prática de arquitetura de software, engenharia de requisitos e desenvolvimento de produtos digitais.

---

## Como Tudo Começou

A ideia surgiu a partir de uma pergunta simples:

> Como garantir que um documento digital não foi alterado ao longo do tempo?

Vivemos em um mundo onde praticamente tudo é digital:

- Contratos
- Certificados
- Relatórios
- Projetos
- Manuais
- Documentação técnica

Mas existe um problema.

Como provar que um documento é autêntico?

Como comprovar que ele existia em determinada data?

Como garantir que ninguém modificou seu conteúdo posteriormente?

Foi nesse momento que comecei a pesquisar sobre:

- Blockchain
- Hashes criptográficos
- Assinaturas digitais
- Smart Contracts
- Sistemas distribuídos

Inicialmente parecia apenas um projeto para aprender blockchain.

Mas rapidamente percebi que estava diante de um problema real.

---

## O Primeiro Grande Aprendizado

No início eu estava fascinado pelas tecnologias.

Rust.

Solidity.

Blockchain.

Smart Contracts.

Python.

Mas logo percebi um erro comum entre desenvolvedores iniciantes:

**Começar pela tecnologia em vez de começar pelo problema.**

O objetivo não era usar blockchain.

O objetivo era criar uma forma confiável de validar documentos digitais.

Essa mudança de mentalidade foi um divisor de águas.

Hoje entendo que boas soluções nascem da compreensão do problema e não da escolha da tecnologia.

---

## Descobrindo a Arquitetura de Software

À medida que o projeto crescia, comecei a perceber que um sistema é formado por várias camadas.

Não basta criar uma tela.

Não basta criar uma API.

Não basta criar um banco de dados.

Tudo precisa funcionar em conjunto.

Foi durante o desenvolvimento do HashDocs que comecei a enxergar arquitetura de software de forma prática.

Perguntas começaram a surgir:

- Onde os documentos serão armazenados?
- Onde o hash será calculado?
- Como validar a autenticidade?
- Como integrar blockchain?
- Como garantir segurança?
- Como reduzir custos?
- Como escalar a solução?

Foi nesse momento que percebi que arquitetura é muito mais do que diagramas.

Arquitetura é tomar decisões.

---

## A Arquitetura do HashDocs

A arquitetura conceitual do projeto é composta por várias camadas trabalhando juntas.

```text
Usuário
    │
    ▼
Frontend Web
    │
    ▼
API Backend
    │
    ├── Banco de Dados
    │
    ├── Serviço de Hash
    │
    └── Smart Contract
            │
            ▼
        Blockchain
```

Cada componente possui uma responsabilidade específica.

Essa separação facilita:

- Manutenção
- Escalabilidade
- Evolução do sistema
- Testabilidade
- Segurança

### Rust

Ela oferece:

- Alta performance
- Concorrência segura

Estudar Rust me fez pensar mais profundamente sobre qualidade e arquitetura de software.

Foi através do Solidity que comecei a entender melhor conceitos como:

## Blockchain Além das Criptomoedas

Uma das maiores descobertas durante o projeto foi perceber que blockchain não serve apenas para criptomoedas.

Ela pode ser utilizada para:

- Certificação documental
- Auditoria
- Rastreabilidade
- Assinaturas digitais
- Prova de autoria
- Registro de propriedade intelectual

No caso do HashDocs, a blockchain funciona como uma camada de confiança.

Ela garante que o registro realizado hoje possa ser validado futuramente.

---

## Carteiras Blockchain

Outro conceito interessante foi a integração com carteiras digitais.

Carteiras como MetaMask permitem que o próprio usuário seja dono de sua identidade digital.

Isso abre portas para funcionalidades futuras como:

- Assinaturas digitais descentralizadas
- Validação de identidade
- Registro de autoria
- Aprovação de documentos

Acredito que as carteiras blockchain terão um papel cada vez mais importante nos próximos anos.

## Engenharia de Requisitos na Prática

Atualmente estou cursando Pós-Graduação em Engenharia de Requisitos de Software.

Mas percebi que muitos conceitos já estavam presentes no HashDocs antes mesmo de estudar formalmente a disciplina.

Antes de implementar qualquer funcionalidade era necessário responder:

- Quem utilizará o sistema?
- Qual problema será resolvido?
- Como o usuário espera utilizar a plataforma?
- Quais informações são realmente necessárias?
- Quais riscos precisam ser mitigados?

Essas perguntas são a base da engenharia de requisitos.

E foi interessante perceber como teoria e prática se conectam.
